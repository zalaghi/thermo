# Thermo

Thermo is a minimal, mobile-friendly temperature dashboard for homelab and small server environments.

It shows live CPU/server temperatures from tiny HTTP agents. It does not use SSH and does not require Grafana, Prometheus, Netdata, Zabbix, LibreNMS, or any other heavy monitoring stack.

## Architecture

Thermo has two small parts:

- Central Thermo app: FastAPI, Jinja2, SQLite, plain CSS, and plain JavaScript.
- Tiny HTTP agents: one read-only FastAPI service on each monitored server.

The central app polls each agent over HTTP with an `X-API-Key` header. Admins add servers through the Agent Setup Wizard, which generates a short-lived pairing token and one-line installer command. Thermo generates the permanent agent API key during setup.

No SSH is used.

## Central App Docker Quick Start

Review `docker-compose.yml` before real use and change these values:

```yaml
THERMO_ADMIN_PASSWORD: change-me-now
THERMO_SECRET_KEY: change-me-to-a-long-random-secret
THERMO_PUBLIC_URL: http://YOUR-THERMO-IP:8088
THERMO_VERIFY_AGENT_ON_COMPLETE: "true"
```

`THERMO_PUBLIC_URL` must be the URL that target servers can reach from your LAN.
You can also set or update it later from `/admin/settings`; the database setting takes precedence over this environment fallback.

Start Thermo:

```bash
docker compose up --build -d
```

Open the dashboard:

```text
http://127.0.0.1:8088
```

The central container listens on port `8080`, the host exposes `8088`, and SQLite is stored in `./data:/data`.

## First Login

Open:

```text
/admin/login
```

Sign in with:

```text
THERMO_ADMIN_USER
THERMO_ADMIN_PASSWORD
```

The compose defaults are `admin` / `change-me-now`. Change the password immediately for real use. Thermo logs a warning when the default admin password or placeholder secret key is used.

## Admin Settings And Branding

Open `/admin/settings` after signing in.

From there you can set:

- App name / display name.
- Public Thermo URL used in generated agent install commands.
- Accent color.
- Favicon.
- Mobile / PWA app icon.
- Optional header icon.

Settings are stored in SQLite in the `app_settings` table. Database settings override environment variables. Uploaded branding files are stored under the Thermo data directory, in `uploads/branding`, and are served from `/uploads/branding/...`.

Accepted upload formats are `png`, `jpg`, `jpeg`, `svg`, `ico`, and `webp`; uploads are limited to 1 MB.

## Add A Server Using The Agent Setup Wizard

Use the wizard as the normal install path:

1. Open `/admin/setup-agent`.
2. Choose the target platform.
3. Enter the server name, agent bind host, agent port, and thresholds.
4. Generate the install command.
5. Copy the command.
6. Run it on the target server.
7. Wait for the wizard status to change to completed.
8. Return to the dashboard.

The wizard token is short-lived, single-use, revocable, and stored hashed. The command does not contain the permanent agent API key. The permanent key is generated by Thermo during bootstrap, written to the target server's local agent env file, and never displayed in full in the admin UI after setup.

The wizard includes an advanced Security options section. When `THERMO_PUBLIC_URL` uses an IP address, Thermo enables "Restrict agent to this Thermo server IP" by default and adds `--allow-client "THERMO-IP"` to the generated installer command. This writes `THERMO_AGENT_ALLOWED_CLIENTS` on the target server so `/temperature` only accepts requests from Thermo.

By default, Thermo verifies the submitted agent URL from the central app before creating the server record. Keep `THERMO_VERIFY_AGENT_ON_COMPLETE=true` unless you are doing isolated local testing.

If the token expires or setup fails, create a new token from the wizard.

## One-Line Installer Examples

Use the command generated by the wizard. These examples show the command shape only.

Recommended secure Proxmox / Debian / Ubuntu command:

```bash
curl -fsSL https://raw.githubusercontent.com/zalaghi/thermo/main/scripts/install-agent.sh | sudo sh -s -- --thermo-url "http://THERMO-IP:8088" --token "PAIRING_TOKEN" --allow-client "THERMO-IP" --rate-limit "120"
```

Recommended secure FreeBSD / TrueNAS CORE command:

```sh
fetch -o - https://raw.githubusercontent.com/zalaghi/thermo/main/scripts/install-agent.sh | sh -s -- --thermo-url "http://THERMO-IP:8088" --token "PAIRING_TOKEN" --allow-client "THERMO-IP" --rate-limit "120"
```

Best-effort Synology DSM command:

```sh
curl -fsSL https://raw.githubusercontent.com/zalaghi/thermo/main/scripts/install-agent.sh | sh -s -- --thermo-url "http://THERMO-IP:8088" --token "PAIRING_TOKEN" --install-dir "/volume1/@appdata/thermo-agent" --allow-client "THERMO-IP" --rate-limit "120"
```

Synology DSM wget alternative:

```sh
wget -q -O - https://raw.githubusercontent.com/zalaghi/thermo/main/scripts/install-agent.sh | sh -s -- --thermo-url "http://THERMO-IP:8088" --token "PAIRING_TOKEN" --install-dir "/volume1/@appdata/thermo-agent" --allow-client "THERMO-IP" --rate-limit "120"
```

Add `--protect-health` if you also want `/health` to require `X-API-Key`.

The installer downloads the agent source tarball from:

```text
https://github.com/zalaghi/thermo/archive/refs/heads/main.tar.gz
```

That GitHub URL is for remote client installation only. Local development should use the local project files.

## Supported Platforms

Thermo's wizard currently targets:

- Proxmox
- Debian / Ubuntu
- Generic systemd Linux
- Fedora / RHEL-family Linux through the generic Linux installer path
- FreeBSD
- TrueNAS CORE
- Synology DSM, best-effort
- Other / Advanced

Linux and Proxmox installs use `/opt/thermo-agent`, `/etc/thermo-agent.env`, and `thermo-agent.service`.

FreeBSD installs use `/usr/local/thermo-agent`, `/usr/local/etc/thermo-agent.env`, and `/usr/local/etc/rc.d/thermo_agent`.

TrueNAS CORE is FreeBSD-based. Service persistence can vary depending on host vs jail and TrueNAS release. Recommended: run the installer inside a FreeBSD jail only if that jail can read the required temperature sensor. If the jail cannot read host CPU temperature, run on the TrueNAS host with care or use a different sensor command.

The default FreeBSD temperature command is:

```sh
sysctl -n dev.cpu.0.temperature | sed 's/C//'
```

If that sysctl is unavailable, the installer stops before registration. Check sensors with:

```sh
sysctl -a | grep -i temperature
```

## Synology DSM

Synology DSM is not a normal systemd Linux environment. Thermo support is best-effort and depends on your NAS model, DSM version, installed packages, and whether the system exposes a readable temperature sensor to user scripts.

Thermo itself does not use SSH. To run the installer command on DSM, use a DSM-supported method such as Task Scheduler user-defined scripts, Container Manager/Docker, or a terminal/SSH session if you choose to enable one.

### Synology With Docker / Container Manager

This is the recommended path when your Synology model supports Container Manager or Docker.

Thermo does not currently publish an official agent image. The wizard therefore shows a Docker build command that downloads the public source tarball and builds a local `thermo-agent:local` image. A truly simple one-line Docker install needs a published image or a Docker-aware installer.

The container must be able to read a temperature source from the NAS. Many models expose `/sys/class/thermal/thermal_zone0/temp`, but not all do. If the container cannot read a temperature, Thermo cannot report it.

### Synology Without Docker

Use the Agent Setup Wizard and select `Synology DSM`.

The non-Docker installer:

- Installs into `/volume1/@appdata/thermo-agent` by default.
- Accepts `--install-dir` if your persistent volume is different.
- Requires existing `python3`, `tar`, and `curl` or `wget`.
- Does not run `apt`, `dnf`, `pkg`, or other package managers on DSM.
- Does not create a systemd service.
- Creates `thermo-agent-run.sh`.
- Creates `synology-task-scheduler-script.sh` for DSM Task Scheduler.
- Tests `/health` and `/temperature` before registering with Thermo.

Recommended DSM Task Scheduler flow:

1. In Thermo, open `/admin/setup-agent` and select `Synology DSM`.
2. Copy the generated Synology command.
3. In DSM, open Control Panel -> Task Scheduler.
4. Create a user-defined script task that runs as root.
5. Paste and run the installer command once.
6. After installation, copy the generated script from `/volume1/@appdata/thermo-agent/synology-task-scheduler-script.sh` into a triggered task that runs at boot.
7. Confirm the wizard status changes to completed.

Manual start command after install:

```sh
/volume1/@appdata/thermo-agent/synology-task-scheduler-script.sh
```

### Synology Limitations

- Not all Synology models expose CPU temperature to user scripts.
- Package availability varies by model and DSM version.
- DSM updates may affect custom startup scripts.
- The installer does not modify DSM firewall rules.
- Firewall/API key restrictions are still recommended.
- If no working temperature command is found, the installer stops and does not register a broken agent.

## Agent Security Model

The Thermo agent is intentionally small and read-only:

- `/temperature` requires the `X-API-Key` header.
- API keys are compared with constant-time comparison.
- `/health` returns only `{"ok": true}` and can be protected with `THERMO_AGENT_PROTECT_HEALTH=true`.
- `THERMO_AGENT_ALLOWED_CLIENTS` can restrict `/temperature` to one IP or CIDR list, for example `192.168.4.7` or `192.168.4.0/24`.
- `THERMO_AGENT_RATE_LIMIT_PER_MINUTE` rate-limits `/temperature` per client IP. The default is `120`.
- The temperature command is read only from `THERMO_TEMP_COMMAND`; request input never affects the command.
- The installer writes the agent env file with mode `600`.

## Firewall Notes

Do not expose agents to the public internet.

Recommended network posture:

- Bind the agent to a LAN/private IP where possible.
- Allow only the central Thermo server IP to reach the agent port.
- Use long random secrets.
- Put Thermo behind HTTPS or a reverse proxy if exposing the dashboard outside your LAN.

Example UFW rule on a Debian/Proxmox agent:

```bash
sudo ufw allow from THERMO-IP to any port 8090 proto tcp
```

Replace `THERMO-IP` with the central Thermo host IP.

## Troubleshooting

Pairing token expired:

- Open `/admin/setup-agent`.
- Create a new token.
- Run the new command on the target server.

Temperature command returns nothing:

- On Debian/Proxmox, run `sensors` and confirm a CPU temperature is visible.
- On FreeBSD/TrueNAS CORE, run `sysctl -n dev.cpu.0.temperature`.
- The installer stores the selected sensor pipeline in `/opt/thermo-agent/read-temperature.sh` on Linux and points `THERMO_TEMP_COMMAND` at that script.
- If your hardware uses a different sensor path, edit `/opt/thermo-agent/read-temperature.sh` and restart the service.

Service does not start:

- Linux: run `sudo systemctl status thermo-agent.service`.
- Linux logs: run `sudo journalctl -u thermo-agent.service -n 80 --no-pager`.
- FreeBSD: run `service thermo_agent status`.
- Check that Python dependencies installed and the configured temperature command returns a number.

Agent shows offline:

- From the central Thermo host, test:

```bash
curl http://AGENT-IP:8090/health
curl -H 'X-API-Key: value-from-agent-env-file' http://AGENT-IP:8090/temperature
```

- Confirm the firewall allows the central Thermo IP.
- Confirm the saved server URL ends with `/temperature`.
- Confirm the agent service is running.

Agent `/temperature` shows Unauthorized in a browser:

- This is expected. `/health` is public, but `/temperature` requires the `X-API-Key` header.
- A plain browser request to `http://AGENT-IP:8090/temperature` has no key and should return Unauthorized.
- Test with the saved agent key instead:

```bash
KEY="$(sudo sed -n 's/^THERMO_AGENT_API_KEY=//p' /etc/thermo-agent.env | head -n1 | sed "s/^['\"]//;s/['\"]$//")"
curl -i http://127.0.0.1:8090/health
curl -i -H "X-API-Key: $KEY" http://127.0.0.1:8090/temperature
```

- If this curl command still returns Unauthorized after reinstalling, restart the service so systemd reloads `/etc/thermo-agent.env`:

```bash
sudo systemctl restart thermo-agent.service
```

Agent `/temperature` returns Forbidden:

- This usually means `THERMO_AGENT_ALLOWED_CLIENTS` is enabled and the request came from an IP that is not allowed.
- On Linux, check:

```bash
sudo sed -n 's/^THERMO_AGENT_ALLOWED_CLIENTS=//p' /etc/thermo-agent.env
```

- Set it to the central Thermo server IP or a safe CIDR, then restart:

```bash
sudoedit /etc/thermo-agent.env
sudo systemctl restart thermo-agent.service
```

- From the central Thermo host, retry:

```bash
curl -i -H "X-API-Key: value-from-agent-env-file" http://AGENT-IP:8090/temperature
```

Central URL not reachable:

- Set `THERMO_PUBLIC_URL` to the URL target servers can reach, for example `http://YOUR-THERMO-IP:8088`.
- Restart the central app after changing the environment.
- Generate a new pairing token so the command uses the corrected URL.

Generated command cannot register:

- Confirm the token has not expired, been revoked, or already been used.
- Confirm the target server can reach `THERMO_PUBLIC_URL`.
- If installation succeeded but registration failed, run the retry command printed by the installer before the pairing token expires.
- From the central Thermo host, test `curl http://AGENT-IP:8090/health` and confirm firewall rules allow access.

Fedora / RHEL-family temperature detection:

- The installer installs `python3`, `python3-pip`, `python3-virtualenv`, `curl`, `ca-certificates`, `tar`, and `lm_sensors` when `dnf` or `yum` is available.
- It only runs `sensors-detect --auto` when supported; it skips interactive `sensors-detect` variants to avoid hanging.
- If setup fails because no temperature was detected, run:

```bash
sensors
sudo systemctl status thermo-agent.service
sudo journalctl -u thermo-agent.service -n 80 --no-pager
```

- If your CPU sensor uses a different label, edit `THERMO_TEMP_COMMAND` in `/etc/thermo-agent.env`, then run `sudo systemctl restart thermo-agent.service`.

## Advanced / Manual Agent Install

The wizard is the recommended path. Manual setup is still useful for unusual systems or testing.

Install agent requirements:

```bash
python3 -m venv .venv
. .venv/bin/activate
pip install -r agent/requirements.txt
```

Run the agent directly:

```bash
export THERMO_AGENT_API_KEY="replace-with-a-long-random-secret"
export THERMO_TEMP_COMMAND="echo 42.5"
export THERMO_AGENT_ALLOWED_CLIENTS="127.0.0.1"
export THERMO_AGENT_RATE_LIMIT_PER_MINUTE="120"
python3 -m uvicorn agent.main:app --host 127.0.0.1 --port 8090
```

Test it:

```bash
curl http://127.0.0.1:8090/health
curl -H 'X-API-Key: replace-with-a-long-random-secret' http://127.0.0.1:8090/temperature
```

For Linux hardware sensors, the default command is:

```bash
sensors | awk '/Package id 0|Tctl|CPU/ {print $0; exit}' | grep -oE '[+-]?[0-9]+(\.[0-9]+)?°C' | head -n1 | tr -d '+°C'
```

Manual server management is available at:

```text
/admin/servers
```

Use it for advanced corrections or deleting servers. Saved API keys are masked in the UI.

## Security Notes

- Admin passwords are hashed with bcrypt.
- Session signing uses `THERMO_SECRET_KEY`.
- Pairing tokens are stored as hashes.
- Pairing tokens expire, are single-use, and can be revoked.
- `/api/status` does not expose agent API keys or pairing secrets.
- Saved agent API keys are masked in the admin UI.
- Agent env files created by the installer are chmod `600`.
- No SSH is required or used.
